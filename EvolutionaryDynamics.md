# Evolutionary Dynamics Plan

## Status and sequencing

Stage 1, finite deterministic Replicator, was completed on 2026-07-15. The
exact-context and policy-contextual prerequisites in
[`RegretMinimization_Review.md`](RegretMinimization_Review.md) are also
complete. Shared particle-core extraction is the next unimplemented stage;
the particle `ReplicatorMutator` remains future work behind that prerequisite.

The plan distinguishes two meanings of continuous:

- finite-strategy replicator dynamics has continuous population shares and a
  continuous-time differential equation, even though its strategy set is
  discrete;
- continuous-trait evolution has an unbounded or continuously valued strategy
  space and needs a distributional approximation such as particles.

## Mathematical scope

For a finite population state `x` and population-dependent fitness `f`, the
replicator equation is

```text
dx_i/dt = x_i (f_i(x) - meanFitness(x)).
```

The implemented exponential step

```text
x_i' = x_i exp(dt * f_i(x)) / sum_j x_j exp(dt * f_j(x))
```

is a positivity-preserving discretization related directly to exponential
MWUA. The evolutionary API remains separate from the online-learning API:
fitness is generated by the current population, there is no sampled
`Choose`/`Learn` interaction, and the current population—not the average
strategy—is the primary state. The implementation is in
[`EvolutionaryBayes/Replicator.fs`](EvolutionaryBayes/Replicator.fs).

For continuous or structured traits, a later particle approximation will
represent the population as weighted trait values. Selection will reweight by
population-dependent fitness; biological mutation will move traits and is not
required to preserve a Bayesian posterior target.

## Staged implementation order

### 1. Finite deterministic `Replicator` — completed 2026-07-15

This stage is item 6 in the current online-learning roadmap.

Implemented surface:

- `Replicator.Population<'Strategy>` is an opaque immutable value over
  distinct named strategies and normalized real-valued shares. `Distribution`,
  `Strategies`, and `Shares` return independent snapshots; `Share(strategy)`
  provides direct population-dependent lookup;
- `Replicator.Fitness<'Strategy>` is
  `Population<'Strategy> -> 'Strategy -> float`;
- `Replicator.create` validates and normalizes caller-owned input masses;
- `Replicator.derivative` is the pure continuous-time derivative;
- `Replicator.step` applies one positive finite time step through the shared
  max-centered log-space reweighting primitive in `MWUA.fs`;
- `Replicator.run` is the sequential repeated-step loop. A zero-step run
  returns an independent population without evaluating fitness.

Completed correctness behavior:

- empty, duplicate-strategy, negative, all-zero, and non-finite populations are
  rejected, as are nonpositive/non-finite time steps and non-finite fitness;
- the probability simplex and zero-share support are preserved. Positive
  support uses a minimum representable unnormalized mass when an exponential
  would otherwise underflow to zero;
- derivative and step are invariant to a common fitness shift and agree with
  hand-calculated updates;
- `run` is exactly repeated `step`, and the extant Go/Stop game approaches its
  stable Go share of `1/101`;
- construction owns its input arrays, public arrays are copies, and pure
  updates do not alter earlier populations.

### 2. Shared particle core extraction

This is now the next unimplemented stage. It has not started.

Before adding continuous-trait evolution, extract reusable mechanisms from
`ParticleFilters.fs` without changing its annealed-SMC behavior:

- log-space weight normalization;
- effective sample size;
- weighted resampling;
- particle mapping and mutation scheduling;
- explicit random-source injection;
- weighted population snapshots and diagnostics.

The Bayesian particle filter remains one consumer of this core. Its existing
prior/likelihood annealing and target-preserving Metropolis-Hastings contract
must remain intact and must have regression tests before the extraction is
declared complete.

### 3. Particle `ReplicatorMutator`

Build a separate evolutionary consumer of the shared particle core for
continuous or structured traits.

Each generation will:

1. evaluate trait fitness against the current weighted population;
2. apply a stable selection reweighting;
3. resample according to an explicit ESS policy;
4. apply biological mutation with an explicit mutation rate;
5. return the next weighted population and diagnostics.

Unlike SMC rejuvenation, biological mutation intentionally changes the
population distribution. It must not be wrapped automatically in an MH
accept/reject step whose purpose is to preserve a Bayesian target.

Exit criteria include deterministic seeded runs, zero-mutation selection,
neutral-fitness mutation, bounded particle count, normalized finite weights,
ESS/resampling behavior, continuous numeric traits, structured traits, and a
comparison against a finite-strategy replicator case where both
representations apply.

## Future work

- Multiple interacting populations for asymmetric evolutionary games.
- Replicator-mutator transition matrices for finite strategies.
- Lineage-aware mutation using the planned persistent lineage representation.
- ESS, diversity, ancestry, invasion-fitness, and fixation diagnostics.
- Evolutionarily stable strategy and local invasion checks.
- Adaptive particle counts or mutation scales.
- Optional trajectory recording, kept separate from any later
  allocation-minimal stepping API.
- Benchmarks comparing deterministic finite dynamics with particle
  approximations on equivalent games.

## Non-goals

- Treating the existing annealed `PopulationSampler` as an evolutionary model
  merely by renaming likelihood to fitness.
- Reusing target-preserving MH mutation as biological mutation without an
  explicit modeling reason.
- Implementing a general PDE solver for continuous population densities.
- Claiming Nash, ESS, or convergence solely because a trajectory reaches a
  numerically stationary point.

## Implementation history

### 2026-07-15 — Finite deterministic Replicator

- Added the opaque immutable `Replicator.Population<'Strategy>`, the
  population-dependent `Replicator.Fitness<'Strategy>` function type, and pure
  `create`, `derivative`, `step`, and sequential `run` functions. Population
  construction normalizes nonnegative masses over distinct strategy names.
- Computed the derivative from centered relative fitness with compensated mean
  summation. The exponential step uses relative log shares and max shifting,
  which makes common fitness shifts irrelevant and avoids overflow. Exact zero
  support is retained; a smallest-representable floor prevents floating-point
  underflow from deleting positive support.
- Kept every population immutable and shallow-owned: caller arrays are copied,
  all public arrays are snapshots, and each returned population has its own
  arrays. Fitness is evaluated against the current population once per named
  strategy on each derivative or step.
- Built the full solution in Release for the library's `net5.0` and
  `netstandard2.1` targets and the `net8.0` test and benchmark projects. Ran
  all 118 tests with zero failures; eight Replicator tests cover normalization
  and ownership, the exact derivative and exponential update, common-shift
  invariance, simplex/support preservation, sequential-run equivalence, the
  Go/Stop stable mixture, and atomic validation. Ran the standalone
  `replicator.fsx` example successfully. No allocation measurement or
  performance benchmark was run.
- The implementation adds duplicate strategy-name rejection and a direct
  `Population.Share` lookup beyond the minimal planned surface. It remains a
  pure finite-strategy module; no shared particle extraction, mutation,
  trajectory recorder, or continuous-trait machinery was started.

### 2026-07-15 — Shared MWUA exponential reweighting correction

- Extracted the max-shifted log-space exponential update into the internal
  `ExponentialWeightsInternals` module in `MWUA.fs`. Stateful MWUA and pure
  finite Replicator now call the same numerical primitive instead of retaining
  duplicate implementations.
- MWUA continues to own persistent log weights, played-strategy averaging,
  sampling, and reset. Replicator constructs log shares for one pure step and
  requests a minimum representable mass for positive support; its derivative
  and population-dependent fitness semantics remain separate.
- Added a direct one-step MWUA/Replicator equivalence test, bringing the full
  Release suite to 119 passing tests. Rebuilt both library targets and the
  `net8.0` projects and reran `replicator.fsx`. No allocation measurement or
  performance benchmark was run.
